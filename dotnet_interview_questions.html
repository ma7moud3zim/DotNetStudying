Day 1
C# Fundamentals
Module 1 — CLR & .NET Basics
Q01 What happens when a C# program runs? ⌄
Answer
  1. You write C# source code.
  2. The compiler converts it to IL (Intermediate Language).
  3. The CLR loads the compiled assembly.
  4. The JIT compiler converts IL into native machine code on the fly.
  5. The CPU executes the native code.
Keywords to mention
CLR IL JIT Assembly
Q02 What is CLR? ⌄
Answer

CLR (Common Language Runtime) is the execution environment for .NET applications. It handles everything so you don't have to manage low-level details.

  • Memory management & Garbage Collection
  • Exception Handling
  • Security enforcement
  • Thread Management
  • JIT Compilation
Q03 What is CTS? ⌄
Answer

CTS (Common Type System) defines all data types shared across every .NET language. It ensures an int in C# is the same type as an Integer in VB.NET.

intdoublestringobject
Q04 What is CLS? ⌄
Answer

CLS (Common Language Specification) is a set of rules that all .NET languages must follow so they can interoperate. For example, a C# library can be consumed directly from VB.NET thanks to CLS.

Q07 What is Boxing and Unboxing? ⌄
Answer

Boxing converts a value type into an object (heap allocation). Unboxing extracts the value type back. Both have a performance cost — avoid in hot loops.

// Boxing — value → object
int x = 5;
object obj = x;

// Unboxing — object → value (must cast explicitly)
int y = (int)obj;

Q10 var vs dynamic — what's the difference? ⌄
Answer

var: type is inferred at compile time — still strongly typed, no runtime overhead.
dynamic: type is resolved at runtime — bypasses compile-time checks, can change type, slower.

var     x = 10;      // int — type fixed at compile time
dynamic y = 10;
y = "Hello";        // OK at runtime — type can change
Module 3 — OOP
Q11 Explain Encapsulation. ⌄
Answer

Bundling data and methods together while hiding internal details. External code only accesses what you deliberately expose through properties or methods — protecting data integrity.

private decimal _salary;

public decimal Salary
{
    get => _salary;
    set => _salary = value > 0 ? value : throw new ArgumentException();
}
Q12 Explain Inheritance. ⌄
Answer

A class can inherit members (fields, properties, methods) from another class using :. C# supports single class inheritance but multiple interface implementation.

public class Animal
{
    public virtual void Speak() => Console.WriteLine("...");
}

public class Dog : Animal
{
    public override void Speak() => Console.WriteLine("Woof!");
}
Q13 Explain Polymorphism. ⌄
Answer

One interface, multiple behaviours. Two types:

  • Compile-time — Method overloading (same name, different parameters)
  • Runtime — Method overriding using virtual + override
Animal[] animals = { new Dog(), new Cat() };
foreach (var a in animals) a.Speak(); // correct method called at runtime
Q14 Explain Abstraction. ⌄
Answer

Hiding the how, exposing only the what. You call db.SaveChangesAsync() without knowing the SQL being generated — that's abstraction. Achieved via abstract classes and interfaces.

Tip: A Dog is an Animal (abstract class). A Dog can ISwim (interface).

Module 5 — Exception Handling
Q19 Why do we use try-catch? ⌄
Answer

To handle runtime errors gracefully instead of crashing the application. catch handles specific errors, finally always runs (great for cleanup like closing DB connections).

try   { var data = await _repo.GetAsync(id); }
catch (NotFoundException ex) { return NotFound(ex.Message); }
catch (Exception ex)         { _logger.LogError(ex, "Unexpected"); return StatusCode(500); }
finally                        { /* cleanup — always runs */ }
Q20 throw vs throw ex — which do you use and why? ⌄
Answer

Always use throw; (no argument). It preserves the original stack trace so you can see exactly where the error started. throw ex; resets the stack trace to the current line — you lose the real origin of the bug.

catch (Exception ex)
{
    _logger.LogError(ex, "Error in GetOrder");
    throw;    // ✅ keeps original stack trace
    // throw ex; ← ❌ loses stack trace
}

ASP.NET Core + EF Core
Module 6 — ASP.NET Core
Q21 Explain the ASP.NET Core Request Pipeline. ⌄
Answer

Every HTTP request passes through a chain of middleware in order, then the response flows back out the same way. Each middleware can short-circuit the chain.

Request → Exception Handling → Authentication → Authorization
       → Routing → Controller Action → Response
// Order matters in Program.cs
app.UseExceptionHandler();
app.UseAuthentication();  // must be before Authorization
app.UseAuthorization();
app.MapControllers();
Q22 What is Middleware? ⌄
Answer

A component that sits in the request/response pipeline. It can inspect, modify, or short-circuit the request. Common examples: authentication, logging, exception handling, CORS.

public async Task InvokeAsync(HttpContext ctx)
{
    // before controller
    await _next(ctx); // call next middleware
    // after controller — e.g. log response time
}
Q23 What is Dependency Injection and why do we use it? ⌄
Answer

Instead of a class creating its own dependencies with new, the DI container provides them via the constructor. Advantages: loose coupling, testability (swap real DB for a mock), and maintainability.

// Without DI — tightly coupled, hard to test
var repo = new OrderRepository(); // can't swap for test

// With DI — injected, easy to mock
public OrderService(IOrderRepository repo) => _repo = repo;

Answer

SingletonOnce for the entire app lifetimeConfig, cache, logging
Q25 Why do we use DTOs? ⌄
Answer

DTOs (Data Transfer Objects) decouple your API shape from your database entity. You control exactly what the client sees and sends.

  • Security — never accidentally expose password hashes or internal IDs
  • Validation — input rules live in the DTO
  • Versioning — change the entity without breaking the API contract
  • Smaller payloads — only send what the client needs
Q26 Model vs Entity vs DTO — what's the difference? ⌄
Answer
TermPurpose
EntityMaps directly to a database table (EF Core uses these)
DTOShapes data for the API — what you send/receive over HTTP
ModelBroad term — sometimes used interchangeably with either
Q27 What is Model Binding? ⌄
Answer

ASP.NET Core automatically reads the HTTP request (route, query string, body, headers) and maps the data to your action method's C# parameters or objects — no manual parsing needed.

[HttpGet("{id}")]
public IActionResult Get(int id) { ... } // id bound from route

[HttpPost]
public IActionResult Create([FromBody] CreateDto dto) { ... } // from JSON body
Q28 What is Routing? ⌄
Answer

Routing maps an incoming HTTP URL + method to a specific controller action. In Web APIs, attribute routing is preferred — you declare the route directly on the controller.

[Route("api/[controller]")]
[HttpGet("{id:int}")]  // → GET /api/products/5
public IActionResult GetById(int id) { ... }
Q29 Attribute Routing vs Conventional Routing? ⌄
Answer

Attribute routing declares routes directly on actions with [Route], [HttpGet] etc. — precise, readable, preferred for APIs.
Conventional routing uses a global template (e.g. {controller}/{action}/{id}) — common in MVC with views.

Q30 What are Filters in ASP.NET Core? ⌄
Answer

Filters run code at specific points in the action pipeline — before or after an action executes. More targeted than middleware (they run only for controller actions).

  • Authorization filters — run first, can block the request
  • Action filters — run before/after action method
  • Exception filters — handle unhandled exceptions from actions
  • Result filters — run before/after the response is written
  • Resource filters — run after auth, can short-circuit (caching)
Module 7 — Entity Framework Core
Q31 What is DbContext? ⌄
Answer

The primary class in EF Core. It represents a session with the database — manages connections, tracks changes to entities, and wraps SaveChangesAsync() to persist all changes at once.

Q32 What is DbSet? ⌄
Answer

A property on DbContext that represents a single database table. You query and modify data through it.

public class AppDbContext : DbContext
{
    public DbSet<Product> Products => Set<Product>();
}
Q33 What is Code First in EF Core? ⌄
Answer

You write C# entity classes first, then EF Core generates the database schema from them (via migrations). Opposed to Database First where you reverse-engineer C# from an existing database.

Q34 What are Migrations? ⌄
Answer

Versioned snapshots of schema changes. Each migration has an Up() (apply) and Down() (rollback). They let you evolve the database without losing data.

# CLI commands
dotnet ef migrations add AddProductTable
dotnet ef database update
Q35 Tracking vs No Tracking — what's the difference? ⌄
Answer

Tracking (default): EF Core watches entities for changes — needed for Update/Delete. No tracking: skips change detection — faster for read-only queries.

// Read-only — faster, less memory
var products = await _db.Products.AsNoTracking().ToListAsync();
Q36 Lazy Loading vs Eager Loading? ⌄
Answer
Lazy LoadingEager Loading
Related data loaded on first accessRelated data loaded with the main query
Can cause N+1 query problemOne SQL query with JOIN
Requires virtual navigation propertiesUses .Include()
// Eager loading — preferred
var orders = await _db.Orders.Include(o => o.Items).ToListAsync();
Q37 Why do we use the Repository Pattern? ⌄
Answer

Abstracts data access behind an interface so business logic never touches EF Core directly. Benefits: easy to unit test (swap real DB for in-memory mock), single place to change if you switch databases.

Q38 What does SaveChanges() do? ⌄
Answer

Generates and executes the SQL (INSERT / UPDATE / DELETE) for all tracked changes since the last SaveChanges call. Call it once at the end — it wraps everything in a single transaction.

Q39 LINQ vs raw SQL — when would you use each? ⌄
Answer

Use LINQ for standard CRUD — compile-time checked, refactoring-friendly. Use raw SQL (via FromSqlRaw) for complex queries, stored procedures, or when LINQ generates inefficient SQL.

Q40 Explain LINQ — what is it and name common methods. ⌄
Answer

LINQ (Language Integrated Query) lets you query collections and databases using C# syntax. It's deferred — the query doesn't run until you call ToList() or iterate.

var result = products
    .Where(p => p.Price > 100)
    .OrderBy(p => p.Name)
    .Select(p => new { p.Name, p.Price })
    .ToList();

products.GroupBy(p => p.Category);
products.FirstOrDefault(p => p.Id == 5);
products.Any(p => p.Stock == 0);

Module 8 — SQL
Q41 What does INNER JOIN return? ⌄
Answer

Only rows that have a match in both tables. Rows with no match on either side are excluded.

SELECT o.Id, c.Name
FROM   Orders o
INNER JOIN Customers c ON o.CustomerId = c.Id
Q42 What does LEFT JOIN return? ⌄
Answer

All rows from the left table, plus matched rows from the right. If no match, right-side columns are NULL. Use this to find "customers who have no orders".

SELECT c.Name, o.Total
FROM   Customers c
LEFT JOIN Orders o ON c.Id = o.CustomerId
-- customers with no orders appear with NULL Total
Q43 Primary Key vs Foreign Key? ⌄
Answer
Primary KeyForeign Key
Uniquely identifies a row in its own tableReferences the PK of another table
Cannot be NULLCan be NULL (optional relationship)
One per tableA table can have many FKs
Q44 GROUP BY vs ORDER BY? ⌄
Answer

GROUP BY collapses multiple rows into groups so you can apply aggregate functions (SUM, COUNT, AVG). ORDER BY sorts the result rows — it doesn't group anything.

SELECT Category, COUNT(*) AS Total
FROM   Products
GROUP BY Category
ORDER BY Total DESC
Q45 HAVING vs WHERE — what's the difference? ⌄
Answer

WHERE filters rows before grouping. HAVING filters after grouping — used with aggregate functions like SUM, COUNT.

SELECT CustomerId, SUM(Total) AS Spent
FROM   Orders
WHERE  Status = 'Completed'        -- filter rows first
GROUP BY CustomerId
HAVING SUM(Total) > 1000          -- filter groups after

Q50 Authentication vs Authorization? ⌄
Answer

Authentication = proving who you are (login, JWT token).
Authorization = what you're allowed to do (admin vs. user role).
Authentication must happen before authorization.

Q51 What is JWT? ⌄
Answer

JSON Web Token — a compact, signed token used for stateless authentication. The server signs it with a secret key; the client sends it on every request in the Authorization header. The server verifies the signature — no DB lookup needed.

Header.Payload.Signature
// Sent as: Authorization: Bearer <token>
The payload is Base64 encoded, not encrypted — never store passwords or secrets in it.
Q52 What are Claims? ⌄
Answer

Key-value pairs stored inside the JWT payload that describe the user. The server reads them to make authorization decisions without a DB call.

UserId Email Role Expiry
Q53 What is CORS? ⌄
Answer

Browsers block requests from a different origin (domain / port) by default. CORS (Cross-Origin Resource Sharing) lets your server explicitly allow certain origins. You'll need this the moment your React frontend talks to your .NET API.

builder.Services.AddCors(o => o.AddPolicy("AllowFrontend", p =>
    p.WithOrigins("http://localhost:3000").AllowAnyMethod().AllowAnyHeader()));
Q54 What is Swagger? ⌄
Answer

Auto-generates interactive API documentation from your controllers and DTOs. You can test every endpoint directly in the browser at /swagger. Included by default in dotnet new webapi.

Module 10 — Async Programming
Q55 Explain async/await — why do we use it? ⌄
Answer

Makes I/O-bound operations (DB, HTTP, file) non-blocking. The thread is freed while waiting — so a single server can handle thousands of concurrent requests instead of one per thread.

// ✅ Thread freed while DB query runs
public async Task<Product?> GetAsync(int id)
    => await _db.Products.FindAsync(id);

// ❌ Blocks the thread — can deadlock
var product = GetAsync(id).Result;
Q56 Task vs Thread — what's the difference? ⌄
Answer
TaskThread
High-level abstraction (TPL)OS-level resource
Managed by the thread poolDedicated OS thread
Lightweight, thousands can runHeavy (~1MB stack each)
Use for async/await operationsUse for true CPU-bound parallelism
Module 11 — Design & Architecture
Q57 Explain SOLID principles. ⌄
Answer
  • S — Single Responsibility: a class should have only one reason to change
  • O — Open/Closed: open for extension, closed for modification (add features without touching existing code)
  • L — Liskov Substitution: a subclass must be usable wherever its base class is used
  • I — Interface Segregation: don't force a class to implement methods it doesn't use
  • D — Dependency Inversion: depend on abstractions (interfaces), not concrete classes
Q58 Explain MVC — Model, View, Controller. ⌄
Answer
  • Model — holds data and business rules
  • View — the UI template (Razor pages) — in Web APIs you return JSON instead
  • Controller — receives requests, calls the service/model, returns a response
In ASP.NET Core Web APIs you mainly work with Controllers and DTOs/Models — View is replaced by JSON output.
