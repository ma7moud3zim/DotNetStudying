Object-Oriented Programming
Part 1
Encapsulation
Core concept

Hiding internal state and only exposing what's needed via properties and methods. Protects data integrity — no one can set an invalid age directly. 

public class BankAccount
{
    private decimal _balance; // hidden

    public decimal Balance => _balance; // read-only

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Must be positive");
        _balance += amount;
    }
}
Inheritance
Core concept

A class inherits members from a base class using :. Use base() to call the parent constructor. C# only allows single inheritance (one base class), but multiple interfaces. 

public class Animal
{
    public string Name { get; set; }
    public virtual void Speak() => Console.WriteLine("...");
}

public class Dog : Animal
{
    public override void Speak() => Console.WriteLine("Woof!");
}
Polymorphism
Core concept

Runtime polymorphism: a base-class reference holds a subclass object — the correct overridden method is called at runtime. Key word: virtual / override. 

Animal[] animals = { new Dog(), new Cat() };
foreach (var a in animals)
    a.Speak(); // calls Dog.Speak or Cat.Speak — decided at runtime
Interfaces vs Abstract Classes
⚡ Frequently asked

Key mental model: Abstract class = is a. Interface = can do. A Dog is an Animal; a Dog can ISwim.

Feature Abstract Class Interface
Multiple inheritance ❌ No ✅ Yes (many interfaces)
Can have fields ✅ Yes ❌ No (only props/methods)
Constructor ✅ Yes ❌ No
Default implementation ✅ Yes ✅ C# 8+ default methods
When to use Shared base logic Define a contract/capability 
public interface IRepository<T>
{
    Task<T> GetByIdAsync(int id);
    Task<IEnumerable<T>> GetAllAsync();
}

public abstract class BaseEntity
{
    public int Id { get; set; }
    public DateTime CreatedAt { get; set; }
    public abstract string GetDescription(); // subclass must implement
}

C# Basics
Part 1
Value types vs Reference types
⚡ Frequently asked

Value types (int, bool, struct, enum) live on the stack and are copied on assignment. Reference types (class, string, array) live on the heap — the variable holds a pointer. 

// Value type – copy
int a = 5;
int b = a;  // b is an independent copy
b = 99;     // a is still 5

// Reference type – same object
var list1 = new List<int> { 1, 2 };
var list2 = list1;   // points to same list!
list2.Add(3);          // list1 also has 3 now
ref, out, in

ref: pass a value type by reference (two-way). out: the method must assign it before returning. in: read-only reference (performance, no copy). 

void Double(ref int x) => x *= 2;
bool TryParse(string s, out int result)  // common pattern
{
    result = int.TryParse(s, out int v) ? v : 0;
    return result > 0;
}
const vs readonly

const: compile-time constant, must be a literal, implicitly static. readonly: set at runtime in the constructor, can be an object. 

public class Config
{
    public const string   AppName    = "MyApp";      // compile-time
    public readonly Guid  InstanceId = Guid.NewGuid(); // runtime, set once
}
String vs StringBuilder
⚡ Frequently asked

string is immutable — every concatenation creates a new object. For loops with many concatenations, use StringBuilder (mutable buffer, far more efficient). 

// Bad – creates many string objects in memory
string s = "";
for (int i = 0; i < 10000; i++) s += i;

// Good – single buffer
var sb = new StringBuilder();
for (int i = 0; i < 10000; i++) sb.Append(i);
string result = sb.ToString();
Nullable types

Value types can't be null by default. Append ? to make them nullable. Use ?? (null coalescing) and ?. (null-conditional) to handle nulls safely. 

int? age = null;
int display = age ?? 0;          // 0 if null

User? user = GetUser();
string? name = user?.Name;      // null if user is null
string safe = name ?? "Unknown";

Exception Handling
Part 1
try / catch / finally + custom exceptions

finally always runs (cleanup). Catch the most specific exception first. Create custom exceptions by extending Exception. 

public class OrderNotFoundException : Exception
{
    public OrderNotFoundException(int id)
        : base($"Order {id} not found") { }
}

try
{
    var order = await _repo.GetOrderAsync(id);
    if (order == null) throw new OrderNotFoundException(id);
}
catch (OrderNotFoundException ex)
{
    _logger.LogWarning(ex.Message);
    return NotFound(ex.Message);
}
catch (Exception ex)      // catch-all last
{
    _logger.LogError(ex, "Unexpected error");
    return StatusCode(500);
}
finally
{
    // Runs always — close connections, dispose resources
}

Delegates, Func, Action & Events
Part 1

A delegate is a type-safe function pointer. Action is a void delegate. Func returns a value. Predicate returns bool. Events are delegates with restricted access. 

// Func<input, output>
Func<int, int, int> add = (a, b) => a + b;
int sum = add(3, 4); // 7

// Action – no return value
Action<string> log = msg => Console.WriteLine($"[LOG] {msg}");

// Predicate – returns bool
Predicate<int> isEven = n => n % 2 == 0;

// Events (publish-subscribe pattern)
public class Button
{
    public event EventHandler Clicked;
    public void Click() => Clicked?.Invoke(this, EventArgs.Empty);
}
// Subscribe:
btn.Clicked += (s, e) => Console.WriteLine("Clicked!");

LINQ
⚡ High priority
💡LINQ is deferred — it doesn't execute until you call ToList(), ToArray(), or iterate it. Calling it twice runs the query twice. 
var products = new List<Product>(); // assume populated

// Filter + Project
var names = products
    .Where(p => p.Price > 100)
    .OrderByDescending(p => p.Price)
    .Select(p => p.Name)
    .ToList();

// GroupBy
var byCategory = products
    .GroupBy(p => p.Category)
    .Select(g => new { Category = g.Key, Count = g.Count() });

// FirstOrDefault – returns null if nothing matches
Product? found = products.FirstOrDefault(p => p.Id == 5);

// Any / All
bool hasExpensive = products.Any(p => p.Price > 500);
bool allInStock  = products.All(p => p.Stock > 0);

// Join (in-memory)
var result = products.Join(
    categories,
    p => p.CategoryId,
    c => c.Id,
    (p, c) => new { p.Name, c.CategoryName }
);

async / await
⚡ High priority

async/await is non-blocking I/O. The thread is freed while waiting for DB/network/file — critical for scalable web APIs. Always return Task (void is only for event handlers). 

// ✅ Correct pattern
public async Task<User?> GetUserAsync(int id)
{
    return await _db.Users.FindAsync(id);
}

// Run multiple tasks in parallel
var (user, orders) = await (GetUserAsync(id), GetOrdersAsync(id)).WhenAll();
// or:
await Task.WhenAll(task1, task2, task3);

// ❌ AVOID – blocks the thread
var result = GetUserAsync(id).Result; // can deadlock!
⚠️ConfigureAwait(false): In library code (not ASP.NET controllers), add .ConfigureAwait(false) after every await to avoid deadlocks in sync contexts.

Dependency Injection
⚡ Most asked

Instead of classes creating their own dependencies, the DI container injects them via the constructor. Makes code testable and loosely coupled.

Transient
New instance every request. Safe for stateless services.
Scoped
One instance per HTTP request. Use for DbContext, repositories.
Singleton
One instance for the app lifetime. Caches, configs, loggers. 
// Register (Program.cs)
builder.Services.AddTransient<IEmailService, EmailService>();
builder.Services.AddScoped<IOrderRepository, OrderRepository>();
builder.Services.AddSingleton<ICacheService, MemoryCacheService>();

// Inject via constructor (preferred)
public class OrderService
{
    private readonly IOrderRepository _repo;
    public OrderService(IOrderRepository repo) => _repo = repo;
}
⚠️Captive dependency: Never inject a Scoped service into a Singleton — the Scoped service won't be disposed per-request and can cause bugs.

REST APIs & Controllers
Part 2
Full CRUD controller with DTOs
⚡ Exam favourite

DTOs (Data Transfer Objects) decouple your API shape from your database model. Never expose your entity directly — you control exactly what the client sees. 

public record CreateProductDto(string Name, decimal Price);
public record ProductDto(int Id, string Name, decimal Price);

[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    private readonly IProductService _svc;
    public ProductsController(IProductService svc) => _svc = svc;

    [HttpGet]
    public async Task<IActionResult> GetAll()
        => Ok(await _svc.GetAllAsync());

    [HttpGet("{id}")]
    public async Task<IActionResult> GetById(int id)
    {
        var product = await _svc.GetByIdAsync(id);
        return product == null ? NotFound() : Ok(product);
    }

    [HttpPost]
    public async Task<IActionResult> Create(CreateProductDto dto)
    {
        var created = await _svc.CreateAsync(dto);
        return CreatedAtAction(nameof(GetById), new { id = created.Id }, created);
    }

    [HttpPut("{id}")]
    public async Task<IActionResult> Update(int id, CreateProductDto dto)
    {
        var ok = await _svc.UpdateAsync(id, dto);
        return ok ? NoContent() : NotFound();
    }

    [HttpDelete("{id}")]
    public async Task<IActionResult> Delete(int id)
    {
        var ok = await _svc.DeleteAsync(id);
        return ok ? NoContent() : NotFound();
    }
}

Include (Eager Loading) & AsNoTracking
⚡ Interview question

Include: loads related data in the same SQL query. AsNoTracking: skips change tracking — faster for read-only queries (no need to detect changes for SaveChanges). 

// Load Orders with their OrderItems
var orders = await _db.Orders
    .Include(o => o.OrderItems)
        .ThenInclude(i => i.Product)
    .Where(o => o.UserId == userId)
    .AsNoTracking()           // read-only = faster
    .ToListAsync();

JWT Authentication
⚡ High priority

Authentication = who are you? | Authorization = what can you do?
A JWT has three parts: Header.Payload.Signature. The server signs it — tamper-proof but not encrypted (don't put secrets in the payload). 

// Setup (Program.cs)
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(o =>
    {
        o.TokenValidationParameters = new
        {
            ValidateIssuer           = true,
            ValidIssuer              = config["Jwt:Issuer"],
            ValidateAudience         = true,
            ValidAudience            = config["Jwt:Audience"],
            ValidateLifetime         = true,
            ValidateIssuerSigningKey = true,
            IssuerSigningKey         = new SymmetricSecurityKey(
                Encoding.UTF8.GetBytes(config["Jwt:Key"]))
        };
    });

// Generate a token
var claims = new[]
{
    new Claim(ClaimTypes.NameIdentifier, user.Id.ToString()),
    new Claim(ClaimTypes.Role, "Admin")
};
var token = new JwtSecurityToken(
    issuer: config["Jwt:Issuer"],
    audience: config["Jwt:Audience"],
    claims: claims,
    expires: DateTime.UtcNow.AddHours(1),
    signingCredentials: creds);

// Protect an endpoint
[Authorize]                        // any authenticated user
[Authorize(Roles = "Admin")]      // specific role

Custom Middleware
Part 2

Middleware intercepts every request/response. Common uses: global error handling, logging, rate limiting, request timing. 

public class RequestLoggingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger _logger;

    public RequestLoggingMiddleware(RequestDelegate next, ILogger<RequestLoggingMiddleware> logger)
    {
        _next = next; _logger = logger;
    }

    public async Task InvokeAsync(HttpContext ctx)
    {
        var sw = Stopwatch.StartNew();
        await _next(ctx);             // call the next middleware
        sw.Stop();
        _logger.LogInformation("{Method} {Path} → {Status} in {ms}ms",
            ctx.Request.Method, ctx.Request.Path,
            ctx.Response.StatusCode, sw.ElapsedMilliseconds);
    }
}

// Register:
app.UseMiddleware<RequestLoggingMiddleware>();

SOLID Principles
⚡ Always asked
S — Single Responsibility Principle

A class should have one reason to change. Don't mix business logic, data access, and email sending in one class. 

// ❌ Violates SRP – does too many things
public class OrderManager { void PlaceOrder(){} void SendEmail(){} void SaveToDb(){} }

// ✅ Each class has one job
public class OrderService   { void PlaceOrder(){} }
public class EmailService   { void SendConfirmation(){} }
public class OrderRepository { void Save(){} }
O — Open/Closed Principle

Open for extension, closed for modification. Add new behaviour without changing existing code — use interfaces and polymorphism. 

public interface IDiscount { decimal Apply(decimal price); }

public class ChristmasDiscount : IDiscount
    { public decimal Apply(decimal p) => p * 0.9m; }

public class BlackFridayDiscount : IDiscount
    { public decimal Apply(decimal p) => p * 0.5m; }
// Add new discount types without touching OrderService
L — Liskov Substitution Principle

A subclass must be substitutable for its base class without breaking the program. If Bird.Fly() exists, a Penguin subclass that throws on Fly() violates LSP. 

// ✅ Design around capability, not hierarchy
public interface IFlyable { void Fly(); }
public class Eagle : IFlyable { public void Fly() => ...; }
public class Penguin { } // does NOT implement IFlyable
I — Interface Segregation Principle

Don't force a class to implement methods it doesn't need. Split fat interfaces into focused ones. 

// ❌ Fat interface forces every class to implement everything
public interface IWorker { void Work(); void Eat(); void Sleep(); }

// ✅ Segregated
public interface IWorkable  { void Work(); }
public interface IFeedable  { void Eat(); }
public interface ISleepable { void Sleep(); }
D — Dependency Inversion Principle

High-level modules should not depend on low-level modules — both should depend on abstractions (interfaces). This is what makes DI work. 

// ❌ High-level depends on concrete class
public class OrderService { private SqlOrderRepository _repo = new(); }

// ✅ Depends on interface – swap implementation without changing OrderService
public class OrderService
{
    private readonly IOrderRepository _repo;
    public OrderService(IOrderRepository repo) => _repo = repo;
}

Design Patterns
Part 3
Repository Pattern
⚡ Very common

Abstracts data access behind an interface. Controllers/services don't care whether data comes from SQL, MongoDB, or an in-memory stub — easy to test. 

public interface IProductRepository
{
    Task<Product?>            GetByIdAsync(int id);
    Task<IEnumerable<Product>> GetAllAsync();
    Task                       AddAsync(Product p);
    Task                       DeleteAsync(int id);
}

public class ProductRepository : IProductRepository
{
    private readonly AppDbContext _db;
    public ProductRepository(AppDbContext db) => _db = db;
    public async Task<Product?> GetByIdAsync(int id)
        => await _db.Products.FindAsync(id);
    // ... other methods
}
Singleton Pattern

Ensures only one instance exists. In ASP.NET you get this for free via AddSingleton — rarely implement it manually. 

public sealed class AppConfig
{
    private static readonly Lazy<AppConfig> _instance =
        new(() => new AppConfig());

    public static AppConfig Instance => _instance.Value;
    private AppConfig() {} // private constructor
}
Factory Pattern

Creates objects without the caller knowing the concrete type. Common when you need to switch implementations based on a condition. 

public interface IPaymentGateway { Task ChargeAsync(decimal amount); }

public class PaymentGatewayFactory
{
    public IPaymentGateway Create(string provider) => provider switch
    {
        "stripe"  => new StripeGateway(),
        "paypal"  => new PayPalGateway(),
        _         => throw new NotSupportedException(provider)
    };
}

SQL
Part 3
JOINs — the most tested SQL topic
⚡ Very important
JOIN type Returns
INNER JOIN Only rows that match in both tables
LEFT JOIN All from left + matched from right (NULLs if no match)
RIGHT JOIN All from right + matched from left
FULL OUTER JOIN All rows from both, NULLs where no match 
-- Get all customers with their orders (or NULL if no orders)
SELECT c.Name, o.OrderDate, o.Total
FROM   Customers c
LEFT JOIN Orders o ON c.Id = o.CustomerId
ORDER BY c.Name;

-- Aggregate: total spent per customer, only those > 1000
SELECT   c.Name, SUM(o.Total) AS TotalSpent
FROM     Customers c
JOIN     Orders o ON c.Id = o.CustomerId
GROUP BY c.Name
HAVING   SUM(o.Total) > 1000
ORDER BY TotalSpent DESC;
💡WHERE vs HAVING: WHERE filters rows before grouping. HAVING filters after grouping (use with GROUP BY + aggregates).
Indexes — why queries are slow
Added ★

Without an index, SQL scans every row (full table scan). An index is a sorted lookup structure — dramatically speeds up WHERE/JOIN on large tables. EF Core lets you define them in code. 

-- SQL
CREATE INDEX IX_Orders_CustomerId ON Orders(CustomerId);

// EF Core fluent config
modelBuilder.Entity<Order>()
    .HasIndex(o => o.CustomerId)
    .HasDatabaseName("IX_Orders_CustomerId");

Added Extras — Important Topics Not to Miss
Bonus
IDisposable & using statement (resource management)
Added ★

Unmanaged resources (DB connections, file handles, streams) must be freed. Implement IDisposable and use the using statement — it automatically calls Dispose() even if an exception is thrown. 

// using statement – Dispose called automatically
await using var conn = new SqlConnection(connStr);
await conn.OpenAsync();
// conn.Dispose() called here even if exception thrown above

// Custom disposable class
public class FileManager : IDisposable
{
    private readonly FileStream _stream;
    public FileManager(string path) => _stream = File.OpenRead(path);
    public void Dispose() => _stream.Dispose();
}
Records (C# 9+)
Added ★

Records are immutable reference types with value-based equality. Perfect for DTOs. with creates a copy with some properties changed. 

public record UserDto(int Id, string Name, string Email);

var user    = new UserDto(1, "Alice", "alice@x.com");
var updated = user with { Email = "newalice@x.com" }; // non-destructive copy
bool same   = user == new UserDto(1, "Alice", "alice@x.com"); // true!
Generic Repository + Unit of Work
Added ★

A generic repository avoids writing CRUD for every entity. The Unit of Work wraps multiple repositories in a single transaction (one SaveChangesAsync call). 

public interface IRepository<T> where T : class
{
    Task<T?>            GetByIdAsync(int id);
    Task<IEnumerable<T>> GetAllAsync();
    Task                 AddAsync(T entity);
    void                 Remove(T entity);
}

public interface IUnitOfWork : IDisposable
{
    IRepository<Order>   Orders   { get; }
    IRepository<Product> Products { get; }
    Task<int> CompleteAsync(); // single SaveChangesAsync
}

Global Exception Handling Middleware
Added ★

Catch all unhandled exceptions in one place, return a consistent JSON error response, and log the error. Interviewers love this pattern. 

public class GlobalExceptionMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<GlobalExceptionMiddleware> _logger;

    public async Task InvokeAsync(HttpContext ctx)
    {
        try   { await _next(ctx); }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Unhandled exception");
            ctx.Response.StatusCode  = 500;
            ctx.Response.ContentType = "application/json";
            await ctx.Response.WriteAsJsonAsync(new
            {
                statusCode = 500,
                message    = "An unexpected error occurred."
            });
        }
    }
}

Caching (IMemoryCache & IDistributedCache)
Added ★

IMemoryCache: in-process cache, single server. IDistributedCache: shared across servers (Redis). Cache frequently-read, rarely-changed data. 

// Register
builder.Services.AddMemoryCache();

// Use (cache-aside pattern)
public async Task<IEnumerable<Product>> GetProductsAsync()
{
    if (!_cache.TryGetValue("products", out IEnumerable<Product>? products))
    {
        products = await _repo.GetAllAsync();
        var opts = new MemoryCacheEntryOptions()
            .SetAbsoluteExpiration(TimeSpan.FromMinutes(10));
        _cache.Set("products", products, opts);
    }
    return products!;
}
